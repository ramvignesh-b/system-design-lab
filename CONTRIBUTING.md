# Contributing

Thanks for helping improve this distributed systems learning repo.

## Ground Rules
- Keep changes focused and explain the learning value.
- Update documentation with any behavior changes.
- Prefer small PRs with reproducible steps.

## Development Workflow
1. Fork or branch from latest `main`.
2. Make changes in one lab or one docs area at a time.
3. Validate commands in your README updates.
4. Open a pull request using this format:
   - What changed
   - Why it matters
   - How to run/verify

## Commit Style
Use semantic prefixes:
- `docs:` documentation changes
- `feat:` new lab behavior or new lab
- `fix:` bug fixes
- `chore:` maintenance and cleanup

## Documentation Requirements
If you touch a lab, ensure its `README.md` includes:
- Quick start
- Verification steps
- Failure scenarios
- Trade-offs and lessons learned

## Testing Expectations
Run relevant checks for changed labs and include outputs or notes in PR.

## First Contribution Ideas
- Improve one troubleshooting section.
- Add one failure scenario experiment.
- Add architecture notes for one lab.
